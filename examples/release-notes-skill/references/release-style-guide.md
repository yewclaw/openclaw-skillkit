# Release Style Guide

## Tone
- Lead with user value, not implementation detail.
- Prefer plain language over internal terminology.

## Structure
- Headline: one sentence, outcome-first.
- Bullets: concrete changes that a customer can notice.
- Impact: who gets it, when, and any rollout caveats.

## Do Not
- Mention internal project names or speculative future work.
- Claim a feature is available to everyone unless rollout status is explicit.
